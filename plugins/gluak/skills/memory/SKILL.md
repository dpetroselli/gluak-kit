# memory

Portable project memory that lives **in the git repo**, so any session opened from any
clone / checkout / pull on any machine has full context. The pattern:

- **`CLAUDE.md`** at the repo root — auto-loaded every session. Index + always-true
  essentials + the capture/push workflow. Kept short.
- **`context/`** — one detailed file per topic area, read on demand when a task touches
  that area. This is where depth lives.

The repo is the single source of truth. It is portable by construction: `git clone` /
`git pull` on any machine restores the full knowledge base — nothing lives only in
`~/.claude/` machine-locally.

## When this skill runs

Detect current state, then pick the mode:

1. **Neither `CLAUDE.md` (repo root) nor `context/` exists** → scaffold mode.
2. **`CLAUDE.md` exists but no `context/`** (hand-written, or generated by `/init`) →
   adopt mode.
3. **Both `CLAUDE.md` and `context/` exist** → maintenance mode.

Always confirm which repo you are operating in before writing files.

**Never overwrite an existing `CLAUDE.md`.** If one is already there it is *integrated*
(adopt mode), never replaced. The `templates/CLAUDE.template.md` scaffold is only ever
written when there is no `CLAUDE.md` at all.

## Scaffold mode (no CLAUDE.md, no context/)

1. Create `CLAUDE.md` at the repo root from `templates/CLAUDE.template.md`. Fill in what
   you can infer about the project (name, what it is, key paths, languages); leave the
   `<!-- TODO -->` placeholders for what only the user can answer, and tell them which.
2. Create the `context/` directory with one starter file, `context/overview.md`, from
   `templates/context-file.template.md`.
3. Do the shared setup steps below.

## Adopt mode (CLAUDE.md exists, no context/)

The existing `CLAUDE.md` is the user's — **integrate it, never overwrite it.**

1. Read the existing `CLAUDE.md` in full. Preserve all of its content and structure.
2. Create the `context/` directory. Move genuinely detailed, topic-specific material out
   of `CLAUDE.md` into `context/<topic>.md` files (one per area) — but only material that
   is clearly depth belonging in context; when in doubt, leave it in `CLAUDE.md`.
3. *Append* to the existing `CLAUDE.md` (do not rewrite what is there):
   - a `context/` index table, with a row per file created;
   - the "Knowledge-capture & push workflow" section from `templates/CLAUDE.template.md`,
     if it is not already present.
4. Show the user a diff of what changed in `CLAUDE.md` and confirm nothing was lost.
5. Do the shared setup steps below.

## Maintenance mode (CLAUDE.md and context/ both exist)

1. Read `CLAUDE.md` and its `context/` index table.
2. When durable knowledge emerges in a session (a decision, a new fact, a state change),
   edit the relevant `context/` file. If it is a new topic area, create a new file from
   `templates/context-file.template.md` and add a row to the `CLAUDE.md` index table.
3. Keep the "Current state" section of `CLAUDE.md` in sync as things change.
4. One context file per topic area. `CLAUDE.md` stays lean — index + essentials only;
   depth goes in `context/`.
5. Do the shared setup steps below.
6. Tell the user in one line what was noted and where. Then drop it.

## Shared setup steps (all modes)

- **Keep the skill versioned.** If `.gitignore` ignores all of `.claude/`, change it to
  `.claude/*` plus `!.claude/skills/` — committed skills then travel with the repo while
  machine-specific state stays ignored.
- **Vendor the skill into the repo.** If the `memory` skill is running from a global
  install (`~/.claude/skills/`) and is not yet committed in this repo's
  `.claude/skills/`, offer to copy it in, so it travels with every future checkout.
- **Do not commit.** Leave everything as working-tree changes — see the workflow below.

## The capture & push workflow (applies in all modes)

- **Context updates are autonomous.** Capture durable knowledge into `context/` as soon
  as it appears — do not wait to be asked. This is what stops things being lost between
  sessions.
- **Updates stay as uncommitted working-tree changes.** Never commit or push autonomously.
- **Commit & push are manual**, on the user's explicit instruction only. When they say
  "push", commit *everything pending* (context updates + any code changes) and push.
- Do not nag about uncommitted changes or repeat "it's in queue" each turn — the user
  knows the workflow.

## Conventions

- `CLAUDE.md` auto-loads at session start, so everything in it is read every session —
  keep it lean.
- `context/<topic>.md` is read on demand. The `CLAUDE.md` index table maps a "when to
  read it" cue to each file.
- Each `context/` file starts with frontmatter (`name`, `description`) and holds factual,
  current, durable knowledge for one area.

## Installing on other machines

This skill ships in the `gluak` plugin of the `gluak-kit` marketplace. Once
`/plugin install gluak@gluak-kit` is done on a machine, `memory` is available in every
project on that machine. See `INSTALL.md` for the in-repo vendoring alternative.
